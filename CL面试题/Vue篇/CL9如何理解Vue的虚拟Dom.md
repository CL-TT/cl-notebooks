## 2. 为什么需要虚拟Dom

1. 在vue中，渲染视图就会调用render函数，**这种渲染不仅发生在组件创建的时候，当视图依赖的数据发生改变时，同样会产生重新渲染**，如果是**直接操作真实Dom的话，由于真实Dom的创建，插入，更新等操作会带来大量的性能的损耗，会导致渲染效率的极大降低**。所以使用虚拟Dom来代替真实的Dom，**主要用于解决渲染效率的问题**



## 3. 怎么把虚拟Dom转化为真实的Dom

### 1. 我们知道在组件的首次渲染，和组件依赖的数据更新时，会生成虚拟 Dom，所以我们来比较一下这两个阶段，用虚拟Dom， 和真实Dom渲染哪个效率跟高一点

####   1. 组件的首次渲染阶段 

1. 在一个组件实例首次被渲染时，它先生成虚拟Dom树，然后根据虚拟Dom树创建真实的Dom，并把真实的Dom挂载到页面中合适的位置，此时，每一个虚拟Dom就会对应一个真实的Dom。
2. 其实做一下对比，**在首次的渲染阶段，操作真实的Dom效率可能还会更高一点， 因为虚拟Dom多了一步生成虚拟Dom的阶段他最终还是要把虚拟Dom转化为真实的Dom******

####   2. 组件依赖数据的更新

1. 这时会生成一个新的虚拟Dom，利用diff算法去和旧的虚拟Dom作比较，然后替换掉有改动的虚拟Dom, 然后去改动真实的Dom，这已经是最小范围的去改动真实Dom
2. **所以利用虚拟Dom的优点就体现在此处**，所以这里更能体现出虚拟Dom的好处



## 4. 虚拟Dom和模板的关系

1. vue框架中有一个compile模块(编译模块)，它主要负责将模板转换成render函数，然后经过render函数生成虚拟Dom
2. 编译的过程分为两步：
   1. 将模板字符串转换成AST(抽象语法树)， 这里的模板字符串指的是template，他最终是一个字符串的形式展现
   2. 将AST转换成Render函数

3. 如果用传统的方式引入js, 则**编译时间发生在组件第一次加载时，称之为运行时编译**

4. 如果是在vue-cli的默认配置下，编译发生在打包的时候，称之为模板预编译(注：不仅在npm run build时打包，运行npm run serve同样会打包，只不过没有生成dist文件) ，**预编译能极大地提高运行时的性能**，所以在运行的时候已经不需要编译了，所以vue-cli在打包时会排除掉vue中的compile模快，以减少打包体积

   ```vue
   <template>
       <div id="app">
            你好呀
       </div>
   </template>
   
   1. 在经过npm run build 或者npm run serve之后，  在文件里你根本就找不到template这个元素了，他已经被编译生成了render函数
   
   
   2. 如果你在配置中配置了,并这样写的话，就会报错，报错如下(终于明白为什么会报这样的错误)
   module.exports = {
     runtimeCompiler: false
   }
   
   <script>
       export default{
           template: `<div></div>`
       }
   </script>
   
   1. 只有把模板单独抽离出来写，才会发生模板预编译，
   2. 这种情况只会走运行时编译，因为配置了runtimeCompile为false， 所以运行时有没有compile模块，编译不了，就会报错
   ```

   ![compile](D:\CL武林秘籍\画图\compile.png)

5. **vue最终运行的时候，最终需要的是render函数，而不是模板，因此，模板中的各种语法，在虚拟dom中都是不存在的，它们都会变成虚拟dom的配置**
