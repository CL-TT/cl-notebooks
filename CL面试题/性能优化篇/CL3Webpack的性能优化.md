# webpack的性能优化

## 1. 分类

1. 构建性能（通常指的是开发环境，不是生产环境）
2. 传输性能
3. 运行性能



## 2. 性能优化的方案

### 1. 减少模块解析（构建性能）

1. 首先模块解析需要经过抽象语法树分析，记录依赖，替换require函数等一系列过程

2. 如果某个模块不进行解析，那么该模块经过loader处理后就是最终代码，如果没有loader，那么源代码就是最终打包结果的代码，那么就可以省略以上过程缩短构建时间

3. **哪些模块不需要解析？**

   模块中没有其他依赖 （通常是一些打包好的第三方库）比如说jQuery

4. **解决办法**

   ```js
   module.exports = {
       module: {
           noParse: //
       }
   }
       
   module.noParse是一个正则，被正则表达式匹配到的不会解析该模块
   ```




### 2. 优化loader的性能 （构建性能）

1. **对于某些库不使用loader**, 比如说babel-loader可以转换ES6或者更高版本的语法，但有些库本身就使用低版本的es语法写的，就不需要转换

   ```js
   所以可以使用exclude， include，  来排除或者仅包含需要应用loader的场景
   
   module.exports = {
     module: {
       rules: [
         {
           test: '',
           include: '',
           use: []
         }
       ]
     }
   }
   ```

2. **缓存loader的结果**

   如果某些文件的内容保持不变，经过loader解析之后，可以保存之前的解析结果，让后续的解析直接使用缓存的结果就可以了

   ```js
   可以使用cache-loader, 一般放在数组的首位
   
   pitch的过程，是一个函数有一个返回值，可返回可不返回， 返回的话就直接结束这个过程
   
   module.exports = {
       module: {
           rules: [
               {
                   test: '',
                   use: ['cache-loader']
               }
           ]
       }
   }
   ```

3. **为loader的运行开启多线程**  thread-loader

   thread-loader会开启一个线程池，线程池中包含适量的线程，他会把后续的loader放到线程池中运行以提高构建效率



### 3. 热更新 HMR（构建性能）

1. 热更新并不能降低构建时间，但他能降低代码改动到效果呈现的时间

```JS
1. 当使用webpack-dev-server

代码变动 -> 重新打包 -> 浏览器刷新重新请求所有资源 -> 浏览器执行代码

2. 如果使用了热更新 （浏览器不会重新刷新）

代码变动 -> 重新打包 -> 浏览器仅请求改动的资源 -> 浏览器执行代码
```

使用和原理

```js
1. 更改配置

module.exports = {
    devServer: {
        hot: true
    },
    
    plugins: [
        // 可选
        new webpack.HotModuleReplacementPlugin()
    ]
}

2. 更改代码 index.js里面

if (module.hot) { // 是否开启热更新
   module.hot.accept() // 接收热更新
}


首先这段代码会参与最终的运行， 当开启了热更新，webpack-dev-server会向打包结果中注入module.hot属性， 默认情况下webpack-dev-server不管是否开启热更新，当重新打包后，都会调用location.reload刷新页面，但如果运行了module.hot.accept将改变这一行为， module.hot.accept的作用就是让webpack-dev-server通过这个websocket管道，把服务器更新的内容发送到浏览器

1. 服务器通过这个socket这个管道告知浏览器我更新了
2. 浏览器知道之后因为有这个module.hot.accept的作用不会去reload， 所以交给插件， 然后又会发送一个请求去询问服务器你更新了什么
3. 服务器做出响应，把更新的内容给浏览器

样式热替换可以使用style-loader
```



### 4. 手动分包（传输性能）

1. 为什么要分包

   减少公共代码，降低总体积，特别是一些大型的第三方库，充分利用浏览器缓存

2. 什么时候分包

   多个chunk引入了公共模块，公共模块体积较大或者较少的变动

3. 基本原理

   ```js
   1. 先单独打包公共模块， 比如说jQuery，单独打包，会被打包成为动态链接库，会暴露出一个变量jQuery， 并生成资源清单
   
   2. 根据入口模块进行正常打包， 如果发现模快中使用了资源清单中描述的模块， 那么打包结果就会发生变化，那么jQuery的打包代码就不会出现在这里，只会导出那个暴露的变量
   
   3. 所以这样极大化的减少了文件的体积
   ```

4. 使用过程

   ```js
   1. 单独打包公共模块，暴露变量名， 单独创建一个webpack.dll.config.js
   
   2. 利用DllPlugin生成资源清单
   
   3. 在页面中手动引入公共模块
   
   4. 重新设置clean-webpack-plugin
   
   5. 使用DllReferencePlugin控制打包结果
   ```

### 6. 代码压缩（传输性能，属于单模块体积优化）

1. 为什么要代码压缩

   减少代码体积，破坏代码的可读性，提升破解成本

2. 压缩的工具 （目前主流的有两种，Uglifyjs和Terser） 

   Uglifyjs是一个传统的代码压缩工具，曾经是前端必备的工具，但由于不支持Es6+语法，流行度已经降低

   Terser是一个新的代码压缩工具，支持Es6+语法，因此被很多构建工具内置使用

   

### 7. Tree Shaking (传输性能，属于单模块体积优化)

tree shaking可以移除模块之间的无效代码

1. 背景

   某些模块导出的代码，不一定会被用到，比如说我有一个模块a, 里面有两个方法，a方法和b方法，在index.js里面只用到了a方法 ，那么b方法就会被tree shaking移除， tree shaking就是用于移除不会被用到的导出

2. 使用

   webpack 2就开始使用tree shaking, 只要是生产环境，tree shaking就自动开启

3. 原理

   **webpack会从入口模块出发，去寻找依赖关系，当解析一个模块时，webpack会根据Es6的模块导入语句来判断，该模块依赖了另一个模块的哪一个导出**

   webpack选择Es6模块的导入语句，是因为

   1. 导入导出语句只能是顶层语句
   2. import的模块名只能是字符串常量
   3. import绑定的变量是不可变的

   

   **在具体分析依赖时，webpack坚持的原则就是先保证代码的运行，然后再tree shaking，所以如果你依赖的是一个导出的对象，由于js语言的动态特性，以及webpack还不够智能，为了代码能够正常运行，他不会移除对象中任何的信息，所以我们在写代码时尽量使用export导出，而不使用export default 使用import {}导入**

   **依赖分析完毕之后，webpack会根据每个模块每个导出是否被使用，标记其他导出为dead code,  然后交给代码压缩工具处理**



### 8. gzip （传输性能）

 gzip它是压缩文件的一种算法，本来是客户端和服务端之间的事情，webpack只是在这其中帮了一点小忙

```js
他是这样的一个传输结构

浏览器 =>(通过请求头：Accept-Encoding: gzip, 表示我接收哪几种压缩方式)

服务器 =>(读取这个未压缩文件， 通过某种压缩方式来压缩文件)

服务器 =>(把压缩的文件传给浏览器， 响应头：Content-Encoding: gizp)

浏览器 =>(浏览器解压这个文件)

优点就是，传输效率可以大幅度提升，缺点就是，服务器的压缩需要时间，客户端的解压也需要时间

webpack的作用就是就行了预压缩， 使用这个插件对打包结果进行预压缩，可以移除服务器的压缩时间
```
