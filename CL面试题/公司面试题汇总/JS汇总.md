##### 5. 谈谈你对事件循环的理解

```js
回答：
1. 正常解析js代码
2. 分析这段代码是同步任务还是异步任务
3. 如果是同步任务，就走js主线程，有一个执行栈， 把这个任务的执行期上下文放入到执行栈中，入栈操作， 执行完成之后就销毁自己的执行期上下文
4. 如果是异步任务，在事件表中注册回调函数， 如果外界触发了这个回调函数，那么就会把这个回调函数放入事件队列中
5. 如果执行栈中的任务全部执行完毕，那么就会从事件队列中一个个的取出回调函数放入到执行栈中， 事件队列分为宏队列和微队列， 先微后宏， 产生自己的执行期上下文，执行完成之后销毁

宏队列：计时器结束的回调， 事件回调， http回调等绝大多数异步函数进入宏队列
微队列：Promise产生的回调会进入微队列， MutationObserver是监听某个Dom对象的变化

6. 这样就形成了事件循环
```

##### 9. 谈谈你对函数柯里化的理解

```js
函数柯里化是把接收多个参数的函数变换成接受一个单一参数的函数， 并且返回接受余下参数的新函数的技术（从字面很难理解， 举例子）

function add(x, y){
    return x + y;
}

add(1, 2);  // 结果是3

// 用柯里化来实现
//1. 把多个参数转化成单一参数
function curAdd(x){
    return function(y){
        return x + y;
    }
}

curAdd(1)(2);   //结果还是3

柯里化最大的好处，就是能够把参数复用(把参数固定下来)

比如说有一个ajax这个函数，有三个参数
function ajax(method, url, data);

//我要使用这个ajax函数
ajax('post', 'www.caolei.com', {name: 'caolei'});
ajax('post', 'www.ceokk.com', {name: 'clei'});

// 你会发现，上面这两个函数的调用，明明第一个参数都是相同的都是post方法， 但每一次调用还是传了相同的参数，写起来很不爽

// 用函数柯里化的方式来写， 这样第一个参数就复用了
const postAjax = curAjax('post');

postAjax('www.caolei.com', { name: 'caolei' });
postAjax('www.ceokk.com', { name })


// 封装通用的方法

/**
* fn: 给我传一个处理函数
* args1: 剩余参数，可以传参数，也可以不传
**/
function curry(fn, ...args1){
    //先获取要处理的函数的形参个数
    const fnParamsLen = fn.length;
    
    return function(...args2){
        const allArgs = [...args1, ...args2];
        
        //如果你传的参数已经达到要求了, 那么直接调用处理方法就行了
        if(allArgs.length >= fnParamsLen){
            return fn.apply(this, allArgs);
        }else{
            //如果还没有达到参数个数要求， 先把传入的参数保存起来，递归调用curry方法, 这个递归的出口就是直到你传的参数，达到了处理函数参数的要求
            return curry(fn, ...allArgs)
        }
    }
}

// 有一道笔试题， 实现一个add方法，使计算结果能够满足如下预期 (就可以用上面封装的curry方法)
add(1)(2)(3)(4) = 10;
add(1, 2, 3)(4) = 10;
add(1)(2)(3, 4) = 10;

function newAdd(n1, n2, n3, n4){
    return n1 + n2 + n3 + n4;
}

const add = curry(newAdd);
```



**10.谈谈你对垃圾回收机制的理解**

```js
1. 如果想知道什么是垃圾回收机制，那么我先来解释一下什么是内存泄漏？

程序的运行需要占用内存，当这些程序没有用到时，如果还不释放内存，就会造成内存泄漏
也就是说不再用到的内存，如果没有及时释放，就被称为内存泄漏， 内存泄漏就会导致程序的卡顿等一系列问题

2. 那么再回头理解这个垃圾回收机制

垃圾回收机制，是一种自动内存管理机制，它可以自动的识别不再使用的变量和对象并将它们从内存中清除，以释放内存空间

// 拔高部分

3. 内存的一些理解(运行机制)

我们都知道js的数据类型被分为，基本类型和引用类型，基本类型存在栈内存中，引用类型存在堆内存中，因为基本类型所花销的内存小，引用类型所花销的内存大，这恰恰是存放在两个不同的空间的原因， 引擎需要用栈来维护程序执行时的上下文状态，如果栈空间大了的话所有数据存放在栈空间中，会影响到上下文切换的效率，从而会影响整个程序的执行效率，所以占内存大的数据会放在堆空间中，引用它的地址来表示这个变量


4. 堆内存的分类

新生代内存区
老生代内存区
大对象区
代码区
map区

5. 主要了解新生代与老生代的垃圾回收

新生代中存放生存时间短的对象(通常只支持1-8M的容量)，老生代存放生存时间久的对象(会支持更大的容量)，针对这两个区域V8分别使用两个不同的垃圾回收器，主垃圾回收器负责老生代的垃圾回收，副垃圾回收器负责新生代的垃圾回收

新生代内存回收

新生代采用的是Scavenge算法，它把新生代空间对半分为两个区域，一半是对象区域(from),一半是空闲区域(to), 新的对象会首先被分配到对象空间，当对象空间快写满时，需要执行一次垃圾清理操作，当进行垃圾回收时，先将对象空间中存活的对象复制到空闲空间进行保存，对未存活的进行回收，复制完成之后，对象空间和空闲空间进行角色对换，空闲空间变成新的对象空间，对象空间变成空闲空间(这时候对象空间已经清空了)，这样就完成了垃圾对象的回收操作，同时这种角色调换能让这两块区域无限重复的使用下去。  当一个对象在两次变换中还存在时，就会从新生代晋升到老生代，这一过程被称为对象晋升策略

老生代内存回收

主垃圾回收器负责老生代区的垃圾回收，其中的对象包括新生代晋升的对象和一些大的对象，因此老生代区中的对象有两个特点，对象占用空间大，对象存活时间长， 因为复制大对象的时间长，所以并不会使用Scavenge算法，它采用的是标记-清除进行垃圾回收，清除后的内存空间里面的对象还是不连续的，所以要引入整理， 先标记-再清除-然后整理， 因为老生代区的对象相对较大，虽然采用标记清除比Scavenge快但架不住卡顿问题，因为js是单线程，因此v8讲标记过程分为一个个子标记过程，同时让垃圾回收标记和js应用逻辑交替进行，直到标记阶段完成，这一算法被称为增量标记算法

6 新生代和老生代的对比

新生代垃圾回收是临时分配的内存，存活时间短；老生代垃圾回收是常驻内存，存活时间长
新生代垃圾回收由副垃圾回收器负责；老生代垃圾回收由主垃圾回收器负责
新生代采用 Scavenge 算法；老生代采用「标记-清除」算法
Scavenge 算法：将空间分为两半，一半是 from 空间，一半是 to 空间。新加入的对象会放在 from 空间，当空间快满时，执行垃圾清理；再角色调换，再当调换完后的 from 空间快蛮时，再执行垃圾清理，如此反复
标记-清理-整理：此为两个算法，「标记-清理」算法和 「标记-整理」算法
标记-清理：标记用不到的变量，清理掉
标记-整理：清理完内存后，会产生不连续的内存空间，为节省空间，整理算法会将内存排序到一处空间，空间就变大了
```
