##### 2. 为什么要使用iframe，它有哪些优缺点？

优点

1. 非常简单，使用起来没有任何的心智负担
2. web应用隔离的非常完美，js,css,dom都完全的隔离开来，不会互相影响
3. 而且我们的子项目之间是独立的没有那么强的通信需求

1. 多个子项目会出现很多的冗余代码 （算多个子应用的缺点）
2. 那个iframe里面带遮罩层的弹窗不会基于主项目居中，只会基于自己居中
3. 全局上下文完全隔离内存变量不共享

‌微前端实现子应用免登录效果主要通过是通过单点登录，当时我们有一个统一身份认证的这么一个机制在，用户在一次登录过后无需再次进行登录验证， 会把令牌保存到localStorage里面， 然后子应用去读取这些信息去验证用户的登录状态

##### 3. 为什么使用乾坤框架，以及背后的原理？

当时做这个的时候，我们的团队已经换到另一个子公司下面了，经过调研对比（京东的Micro-app）以及为了和他们那边保持一致最终选择了乾坤

#####    3.1 乾坤是怎么加载子应用？

   乾坤是基于single-spa来实现的，通过在主应用中注册微应用方式（registerMicroApps），把路由path跟子应用关联起来，跳转对应路由时，微应用就会被插入到指定的container（dom）中。同时依次调用微应用暴露出的生命周期钩子

```js
import { registerMicroApps, start } from 'qiankun';

registerMicroApps([
  {
    name: "app-vue2-demo", //子应用名称，唯一
    entry: "//localhost:4001", //子应用地址
    container: "#subapp-container", // 子应用挂载的div
    activeRule: "/app-vue2-demo", //子应用激活规则
  },
]);

// 启动 qiankun
start();
```



京东的micro-app是通过web components来实现的，先在主应用里自己定义路由，然后在路由组件里使用 `Web Components`

```js
// 路由配置
{
  //路由路径最好是非严格匹配
  path: "/app-vue2-demo*",
  name: "Vue2DemoPage",
  component: () => import("@/views/Vue2DemoPage.vue")
}

<template>
  <div>
    <!-- 
      name(必传)：应用名称
      url(必传)：应用地址
      baseroute(可选)：基座应用分配给子应用的基础路由
     -->
     <micro-app 
      name='app-vue2-demo'
      url='http://localhost:4001/' 
      baseroute='/app-vue2-demo'
     >
    </micro-app>
  </div>
</template>
```

## 2. 封装网络库

##### 1. 讲讲你是如何实现请求库的？

方案选择：首先我考虑的是实现必要性，前端本身就存在着诸多的请求库像成熟度比较高的axios，但他们都是一些基础库，没有提供上层功能，比如说请求重试，请求缓存，请求幂等这些都没有，另外像一些上层的请求库倒是有这些功能，比如VueRequest,SWR,但他们都和具体的框架绑定了，并且没有提供请求的基础功能，最重要的一点是市面上的公共库不可能提供基于公司内部协议的封装，所以我们就考虑重新实现基于公司内部业务的通用请求库

技术实现：

1. 我把请求库分为三层，从下到上依次是请求基础库，负责发送请求的基本功能，然后是请求核心库，负责实现各种上层逻辑比如重试，缓存等，最上层是请求业务库，负责在代码层面封装公司的内部协议
2. 在结构上，考虑到请求基础库的具体实现方式可能有变动，axios换成fetch，或者以后出了什么新的技术，为了减少对上层代码的影响，让核心库不依赖具体的请求基础库，核心库仅提供接口（用来描述这个request是什么样子的），请求基础库可以基于接口提供不同的实现方式，在业务库注入具体的实现（只需要去切换不同的引入就可以了）
3. 比如说具体的上层功能，请求缓存，要考虑几个问题，请求的结果怎么存（内存/持久化存储），存在哪，缓存的键是什么， 这些都是可变的， 提供一个仓库接口（has，set,   get）所以就需要用户自己去指定缓存的方案和缓存的键，

##### 请求重试会出现的场景以及策略

1. **网络波动或者中断**

   场景：用户设备网络不稳定的情况下

   去监听这个NetWork Error错误事件，结合指数退避原则重试，首次失败等待1秒重试，第二次2秒，第三次4秒，避免加剧服务端的压力

2. **服务器端短暂的错误**

   场景：服务端返回500（服务器发生错误） 503（服务器过载）， 

   去识别可重试的状态码 502（代理配置错误）， 504（代理配置超时）

3. **请求超时**

   比如说分页加载

   用户主动触发的分页加载，用户点击下一页或者滚动加载更多数据时，请求超时， 可以自动重试1-2次，避免用户反复手动操作

   不适用于那种高频的分页请求





## 3. 封装一个可以保存和同步表单数据的工具库

##### 1.解决什么样的问题

大型的管理系统中，表单页的内容会非常多，就会存在这么几个问题，在填写表单数据的时候数据可能会有丢失的风险，表单数据怎么恢复，多个标签页同一个表单的数据怎么同步，会有这么几个问题，基于这个需求，将核心逻辑封装成一个工具库



##### 2.解决方案

拿到这个需求之后，我先对功能模块进行了归类和拆解，最终整理出来的就是两大块， 数据的自动保存与恢复，多标签页间的数据同步，我考虑的是将这个功能模块封装成一个不受具体框架限制的公共模块，，无论是在vue中还是react中都是适用的，因此我将其设计为类的结构，并且内部的代码不会使用某个框架特定的API



##### 3.具体的技术方案

当时在设计的时候，把它分为两个类，一个是FormStorage类专门负责处理，数据的本地存储，恢复，删除等操作，多标签的这个数据同步，另一个是FormStorageManage这个类他是专门管理FormStorage类实例，之所以要用这个FormStorageManage来管理FormStorage是为了

1. 集中管理，确保所有实例的统一初始化，数据保存，和事件处理
2. 优化事件监听，避免在每一个FormStorage实例上重复绑定全局事件监听器，从而减少不必要的性能开销

存储方案这一块用的localStorage，多标签的数据通信使用的是BroadcastChannel





## 4. 对浏览器计时器节流的处理做了封装

1. ##### 问题背景

   当时有这样的一个需求，如果长时间不对页面进行编辑，就自动帮你退出编辑状态

   

2. ##### 解决方案

   刚开始想的方案是这样的，用户开始编辑的时候，启动一个倒计时，每次用户有操作就清除之前的倒计时，如果用户处于非活动状态（通常是监测用户的活动事件）如果在指定的时间内没有触发这些事件，那么就退出编辑状态

   一开始使用setInterval进行计时，功能这些都没什么问题，但是测试的时候偶尔发现时间不太准确，因为它的运行机制（事件循环，可能会被延时执行，最小的延迟时间为4毫秒）时间有些许偏差是可以接收的，但是测试发现的这个不准确是非常夸张的，最后发现问题发生在当前页面失活的情况下，浏览器可能会降低计时器的执行频率以节省资源，导致这个延迟时间被延长‌



3. ##### 实际的解决方案

   所以为了解决这个问题，使用了web worker, 把这个计时器直接放到另外的线程中，通过主线程与web worker之间进行通信，发送消息来维持web worker线程中计时器的开关

   但是这样处理，对于开发其实是有心智负担的，我们的目的就仅仅是通过计时器做简单计时。但是为了解决失活标签页计时器节流的问题，不得不把计时器放入到`WebWorker`线程，而计时器的开关，我还得通过发送开关消息到`WebWorker`线程进行处理。

   因此我把`WebWorker`线程做了封装，通过动态生成（( URL.createObjectURL（blob））`WebWorker`的脚本，在主线程劫持模拟`setTimeout`与`setInterval`，在`WebWorker`线程中监听主线程发送的信息，并执行对应的计时器，然后在主线程监听消息，并做出对应处理即可。



## 5. TTS性能优化 （文字转语音）

TTS其实就是文本转语音的这样的一个功能，

1. ##### 背景

​      它的背景是这样的，当时智慧校园服务里面有这么一个场景，一些老师会分享一些知识文档，然后文档支持在线语音播放的这样的一个功能



2. ##### 遇到的问题以及解决方案

由于 WebAPI 的语音转换太呆板了，所以我们当时选择的是讯飞的 TTS 服务，不过会遇到这么一些问题 

1. 文本过大导致转换时间过长 
2. 转换出来的音频数据过大，传输也比较耗时
3.  需要根据文本内容给语音添加情绪

我主要就是针对这些问题来进行优化。首先第一个问题和第二个问题其实是同一个问题，文本过大，自然传输到 TTS 服务所花费的时间就会越久，那么自然生成的音频数据也就越大，而且这一块儿用的是第三方服务平台，我是没有办法下手的。所以我就想到了从文本方面下手

1. ##### 进行文本切割 （把这个大文本，切割成一份份的小文本， 然后去分段请求）

   有两种方式，第一种是按照标点符号断句然后切割， 第二种是按照固定长度字符进行切割，在项目中我使用的是第一种方式，因为这样能保证句子的完整性，不会切断语义，而且也方便后面做这个情绪标记，

   

   不过文本切割后会带来这个文本数量变多的问题， 这里我采用的是并发的方式来处理的

   

2. ##### 并发控制

   并发控制这边我主要做了这么几件事情： 

   1. 限制并发数量，这样能控制到达服务器的请求数量 

   2. 使用队列管理和调度并发请求，确保新的请求在当前并发请求完成后才发出的 

   3. 根据服务器的负载情况和网络状态，动态的调整最大并发请求数

      这样就达到了并发发送多个文本的目的，同时又能对并发量进行一个很好的控制

   4. 另外在“发送”这一块儿，我还做了一层优化，就是请求的取消。因为我考虑到这么一种情况，就是用户鼠标先 点击一段文本，本来要对这段文本进行转换的，但是用户又点击到了其他文本，那么第一次的请求就不需要了，这里可以将其取消掉。

      

3. ##### 请求取消

   就讲这个AbortController



4. ##### 缓存优化

   响应回来的数据也是可以通过缓存来优化的， 当时我在考虑缓存的时候，也是考虑了两种方案，一种是缓存在客户端，另一种是缓存在服务器端

   缓存在客户端的好处在于获取缓存的速度很快，可以采用 localstorage 来存，值存对应的 base64 格式的数据，键则是对应的文本内容。不过这里有个细节的点要考虑，就是键如果直接用文本的话，可能一句话过长，导致键超出 localstorage 的长度限制，所以需要做一层 MD5 处理。本来都打算这么做了，后来我想了一下，客户端的缓存只能存当前用户转换过的文本，但是一般来讲，用户看完某篇知识文档后，转而就会去看下一篇，不太会一篇文档翻来复去听个十来遍，除非是一篇超高质量、需要反复玩味的文档。

    所以我改变了这个想法，跟后端商量了一下， 决定让他缓存在这个服务器端，这样只要转换过某个文本，都能缓存起来，方便下一次使用

   

5. ##### 情绪标记

   至于添加情绪标记，这个前端不太好处理，涉及到自然语言的处理，而 python 是这一块儿的强项，所以就和后端的同事商量，看他能不能搞定这一块儿。最终他还是搞定了，返回给我的是音频流数据。 我拿到音频流数据后，转换成 base64 格式的数据，这样更方便页面上的 audio 元素使用



整套优化方案落地后，以前的一个长文本平均需要 30s 的转换时间，优化后，总时间降到几秒以内，如果是存在缓存的情况下，可能更快一点。

 不过现在回想起这个项目，我认为缓存这一块儿仍然有优化的地方，就是可以设置双层缓存，客户端用 localstorage 缓存一次，客户端找不到，就去后端数据库去找，如果还是找不到，最后再请求 TTS 服务，不知道这种方案会不会更好一点

## 7. 在没有登录的情况下，如何确认访问者是谁？

##### 1. 背景

当时做这个博客项目的时候，博客文章是可以点赞的，那么我就需要去来记录他是谁， 那么下次他在访问时候可以展示出他点赞的状态



##### 2. 方案

1. 根据IP地址来判断

   优点：非常简单，服务器可以直接获取

   缺点：极其不准确

   1. 动态IP：大多数家庭用户的IP地址由ISP动态分配，会经常变化。

   2. NAT：多人会共享一个公网IP, 你无法区分是哪一个人

   3. vpn/代理：这些都是可以伪装自己的真实ip的

   4. 或者还有一些可以伪造ip地址的  （use-agent）

      

2. 浏览器指纹（Fingerprint.js）:  它通过收集浏览器和设备的各种特征信息来生成一个唯一标识。

   优点： 持久性（就算你清楚cookie，使用无痕模式，都没有什么影响，  因为它是硬件和软件的特性）

   不需要客户端存储

   缺点： 有隐私争议

   

3. uuid （我在项目中使用的是这种方式）

1. 用户初次访问服务器时候，就会在服务器生成一个uuid,  然后通过cookie的形式返回给浏览器， 那么这个uuid就是该访问者的唯一标识

​    优点：就是使用起来非常简单， 在测试的时候很方便

​    缺点：清楚cookie，就会导致这个标识失效
