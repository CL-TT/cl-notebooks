##### 1. 在换包的时候，浏览器有缓存，可能页面没有更新，你有什么解决办法吗？

1. **提示用户清除缓存**让页面强制刷新，这种方式简单粗暴，但是需要用户配合，可能会影响用户的体验

2. **在http响应头中使用cache-control字段来控制缓存行为**，public（服务器资源是公开的）private（服务器的资源是私有的）no-cache（告诉客户端，你可以缓存这个资源，但是不要直接使用它，当你缓存之后，后续的每一次请求都需要附带缓存指令，让服务器告诉你这个资源有没有过期）no-store（告诉客户端，不要对这个资源进行任何的缓存，之后的每一次请求都按正常的普通请求进行，浏览器不会对该资源进行任何的缓存处理）

3. **不缓存入口文件**，在入口文件head中为meta标签来添加不缓存的标识，来控制缓存行为

   ```html
   <meta http-equiv="pragram" content="no-cache">
   <meta http-equiv="cache-control" content="no-cache, no-store, must-revalidate">
   <meta http-equiv="expires" content="0">
   ```

   

4. 版本控制,动态版本号，打包的时候让文件名，带版本号和时间戳的





##### 2. 在发送一个请求时，怎么放弃这个请求（请求重试）？

```js
1. 原生的ajax终止请求

可以使用XMLHttpRequest.abort()，用于终止这个对象的请求


2. axios终止请求

2.1 （这种方式也是适用于fetch api的） 浏览器原生的这个AbortController来终止请求, 通过new这个对象得到实例，通过这个实例创建一个信号， 然后在这个请求的配置里面去绑定这个信号， 然后可以在任何时候去调用实例中的abort方法

const abortController = new AbortController()

const signal = abortController.signal


2.2 cancelToken (但这种方式好像被弃用了)

首先创建一个取消标记， 然后这个取消标记来关联请求，然后再调用这个cancel方法
const source = axios.CancelToken.source()

axios.post('', { cancelToken: source.token })
```





##### 3. hash和history模式下，反复刷新404怎么解决？

主要因为，浏览器在输入路径按下回车的时候，会向服务器发送一次get请求，但是由于服务器并没有去匹配这个请求的响应，所以就会出现404的情况， 那么解决办法也很简单，就是当你访问到那些不存在的路径，让它都指向入口文件index.html就可以了

```js
Nginx

location / {
    try_files $url $uri/ /index.html
}
```





##### 4. 如何解决页面请求接口大规模并发的问题

前端页面同时发起了很多的接口请求，那么就会导致性能问题，或者后端的压力过大，所以就要考虑这个并发控制的策略，会有这几种方法

```js
1. 请求队列化，promise并发控制， 通过这个队列机制，控制同时发起的这个请求数量  (讲解自己封装的方法concurRequest)


2. 分页与分片

@1. 按需加载：将大数据拆分为分页或者分片请求

@2. 滚动加载：仅渲染可视区域内的数据，减少无效的请求

3. 防抖与节流， 高频操作（比如说搜索框输入）进行这个请求合并

4. 缓存策略

@1. 内存缓存：短期缓存重复的内存结果（Map, weakMap）

@2. HTTP缓存：利用这个catch-control减少这个重复的请求

@3. Service Worker: 离线缓存或者预加载资源
```



##### 5. js执行100万个任务，如何保证浏览器不卡顿

因为js它是单线程语言，在执行大量的任务时，确实会导致浏览器卡顿或者响应变慢，为了浏览器的性能不受影响，可以采取以下几种策略

先讲自己模拟的场景

```js
1. 任务分片 (讲解自己封装的方法runTask)

将大任务分成多个小任务，分批次来处理， 每次只处理一部分任务，然后通过这个setTimeout分批次处理任务, 使用requestAnimationFrame在每一帧空闲的时间来执行任务，或者requestIdleCallback方法也可以


2. web worker

web worker是创建一个独立的线程来执行任务，这个线程与主线程互不干扰

const worker = new Worker()

worker.postMessage()

worker.onmessage = () => {
    
}
```



##### 6. 长列表如何优化

```js
1. 分页加载

分页加载应该是一个基本的方法，把数据分成多个页面，每次只加载一页的数据，减少一次性渲染的负担，但这种方式不适用那种需要无限滚动的场景

2. 懒加载

懒加载的原理：一开始只会加载可视区域内的数据，其他数据在用户滚动到底部时才会被加载，这样可以显著减少初次加载的时间，提高页面的响应速度

缺点：懒加载只能应对数据不是太多的情况，如果列表项有几万或者几十万，那么最终会有对应数量的DOM存在于页面上，严重的降低了页面的性能

3. 时间分片(暂不说)

4. 虚拟列表

虚拟列表的原理：设置一个可视区，然后用户在滚动列表的时候，本质上是动态修改可视区域里面的内容， 动态的去改变可视容器的Y轴偏移量

比如说一个可视区域内有5个项目， 不管用户怎么滚动， 页面上对应的DOM数量始终是5个
```
