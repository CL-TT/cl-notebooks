# CSRF攻击与防御

## 1.CSRF的特点和原理

CSRF：Cross  Site  Request  Forgery    跨站请求伪造

本质：**恶意网站把正常用户作为媒介，通过模拟正常用户的操作，攻击其登录过的站点**  



原理如下：

1. 用户访问正常站点，登录后，用户获取到了正常站点的令牌，以cookie的形式保存

   ![CSRF1](D:\CL武林秘籍\画图\CSRF1.png)

2. 用户访问恶意站点，恶意站点通过某种形式去请求了正常站点（请求伪造）迫使正常用户把令牌传递到了恶意站点，完成攻击

## 2. 如何防御

### 1.cookie的SameSite

现在很多浏览器都支持禁止跨域附带的cookie，只需要把cookie设置的sameSite设置为strict即可

sameSite有如下值

1. strict：严格， 所有跨站请求都不附带cookie，有时会导致不好的用户体验
2. Lax：宽松，所有跨站的超链接，get请求的表单预加载连接时会发送cookie，其他情况则不会
3. none：无限制

这种方法非常简单，及其有效，但用户不能使用太旧的浏览器



### 2.验证Referer和Origin

页面中的二次请求，都会附带referer或者origin请求头，向该服务器表示该请求来自于哪一个源或者页面，服务器可以通过这个头进行验证

但某些浏览器的referer是可以被用户禁止的，尽管这种情况很少

但是他无法识别base64格式的， 就不会有referer这个字段



### 3.使用非cookie令牌

就是需要每一次请求在请求体或者请求头中附带token

比如说 authorization：token



### 4.验证码

每一次的CSRF请求都需要填写验证码， 这样会把用户逼疯



### 5.表单随机数

这种情况是针对服务器端渲染而言的

流程

1. 客户端请求服务器，请求添加学生的页面，传递cookie

2. 服务器

   1. 生成一个随机数，放到session中

   2. 生成页面时，表单中加入一个隐藏的表单域

      <input></input>

   3. 填好信息，提交表单，会自动提交隐藏的随机数

   4. 服务器拿到cookie，判断是否登录过

   5. 对比提交过来的随机数和之前的随机数是否一致

   6. 清除掉session中的随机数



### 6.二次验证

做出敏感操作的时候，做二次验证
