# PNPM的原理

## 1. 先要弄清楚几个概念

### 1. 文件的本质

1. 在操作系统中，文件实质上是一个指针，只不过他指向的不是内存地址，而是一个外部存储地址（这里的外部可以是硬盘，u盘，甚至是网络）

![pnpm1](D:\CL武林秘籍\画图\pnpm1.png)

2. 所以当我们删除文件时，删除的实际上是指针，因此，无论删除多么大的文件，速度都非常快



### 2. 文件的拷贝

如果你复制一个文件（A），是将该文件的指针指向的内容进行复制，然后产生一个新文件指向新内容（B）， 所以改变A，B并不会变化

![pnpm3](D:\CL武林秘籍\画图\pnpm3.png)

### 3. 硬链接 hard link

硬链接的概念来自于Unix操作系统， 它是将文件A的指针复制到另一个文件B的指针中， 文件B就是文件A的硬链接， （**两个指针指向了同一个地址， 所以改变文件A里面的内容， 文件B里面的内容也会随之改变**）

**最重要的一点就是， 和复制不同的是，他不会产生额外的磁盘占用，文件B是不占磁盘空间的**



1. 硬链接的数量是没有限制的，可以为同一个文件产生多个硬链接
2. windows系统也支持了创建硬链接的操作，只能在cmd中 使用命令  **mklink  /h  新文件名称  目标文件（链接谁）**暂时还区分不了谁是硬链接
3. 因为文件夹不存在内容， 所以文件夹不能创建硬链接
4. 种种原因， 在windows系统中，通常不要跨越盘符创建硬链接

![pnpm4](D:\CL武林秘籍\画图\pnpm4.png)

### 4. 符号链接（软链接）symbol link

符号链接又被称为软链接， 如果为某个文件或者文件夹A创建符号链接B， 则B指向A

![pnpm5](D:\CL武林秘籍\画图\pnpm5.png)

1. windows系统支持符号链接的创建， 在cmd中使用  **mklink  /d   新文件名称    目标文件** 

   **/d表示创建的是目录的符号链接，不写则是文件的符号链接**



### 5. 硬软链接的区别

1. 硬链接仅能链接文件， 而符号链接可以链接目录
2. **硬链接在链接完成之后，仅和文件内容关联**， 和之前链接的文件没有任何关系， **而符号链接始终和之前链接的文件关联**，和文件内容不直接相关
3. node对硬链接的处理， node不对其做任何特殊处理，也无法区别对待，实际上node根本无从知晓该文件是不是一个硬链接
4. node对软链接的处理， 由于符号链接指向的是另一个文件或者目录，当node执行符号链接下的js文件时，会使用原始路径



## 2. PNPM原理

```js
用一个例子来说明它的构建，   假设有两个包a和b,   a依赖b
```

![pnpm6](D:\CL武林秘籍\画图\pnpm6.png)

```js
假设我们的工程叫cl-proj, 这个工程只依赖了a包， 则在安装时， pnpm会做如下处理

1. 查询依赖关系， 得到最终要安装的包  a和b  (b包是因为a依赖了b)
2. 查看a和b是否已经有缓存，如果没有下载到缓存中，如果有则进行下一步
3. 创建node_modules目录， 并对目录结构初始化
```

![pnpm7](D:\CL武林秘籍\画图\pnpm7.png)

4. 从缓存的对应包中使用硬链接放置文件到相应包代码目录中

![pnpm8](D:\CL武林秘籍\画图\pnpm8.png)

5. 使用符号链接，将每一个包的直接依赖放置到自己的目录中

![pnpm9](D:\CL武林秘籍\画图\pnpm9.png)



1. 为什么在a包的1.0.0文件夹下会出现node_modules?

   因为，在执行a代码的过程中，可以读取到他的直接依赖

2. 新版本的pnpm是怎么解决，一些不规范的包（就是读取了间接依赖）?

   它将所有工程的非直接依赖，使用符号链接加入到了  .pnpm/node_modules,  至此解决了自身的缺点， 变成了一个非常完美的包管理器

##### 6. 但是如果b依赖c，a又要直接用c，这种不规范的用法现在pnpm通过这种方式支持了， （但是那些使用绝对路径的写法，可能永远也无法支持了）

![pnp10](D:\CL武林秘籍\画图\pnp10.png)





## 3. PNPM解决的问题（yarn和npm存在的问题）

1. ##### node_modules目录变的更加简洁？

   pnpm在node_modules目录下只展示我项目的直接依赖 （这个是我一直想吐槽的，每次想看这个项目有哪些依赖，一打开一大堆，pnpm解决了这个问题）， 那些间接依赖放在别处

   

2. ##### 当多个项目使用缓存中的包时，不占磁盘空间？

   yarn和npm是把缓存中的包复制到项目中的，  磁盘空间还是占据的， **pnpm通过硬链接那些项目中的包指向的还是同一个磁盘空间，所以节省了大量的磁盘空间**
